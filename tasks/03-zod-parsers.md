# Part 3: Zod Parsers

Implement runtime validation using Zod, exported from `@storyblok/schema/zod`. Zod schemas are auto-generated from OpenAPI specs (via Part 1's `generate.ts`) and wrapped with ergonomic `parse*` functions that mirror the `define*` API from Part 2.

## Acceptance criteria

- `import { parseStory, fieldSchema, storySchema } from "@storyblok/schema/zod"` works when `zod` is installed.
- `parseField(...)` validates a field config and throws `ZodError` on invalid input.
- `parseStory(component, story)` dynamically validates story content against the component schema and throws `ZodError` on wrong types, extra keys, or missing required fields.
- `parseStory(component, story, { withDefaults: true })` fills missing optional fields with `default_value` from the component schema.
- Exported schemas (e.g., `fieldSchema`) can be composed with user's own Zod schemas.
- `pnpm --filter @storyblok/schema test:unit:ci` passes all Zod-related tests.
- `pnpm --filter @storyblok/schema test:types` passes (parse function return types match the TypeScript types from Part 2).

---

## Prerequisite

Parts 1 and 2 must be complete:
- `src/generated/zod-schemas.ts` exists (produced by `generate.ts`) and exports raw Zod schemas from OpenAPI.
- Types from Part 2 (`Field`, `Component`, `Story`, `Asset`, `Datasource`, etc.) are available in `src/types/`.

## 3.1 Generated Zod schemas — `src/generated/zod-schemas.ts`

This file is auto-generated by `generate.ts` (Part 1) using `openapi-zod-client`. It should not be edited manually.

Expected exports (names may vary based on `openapi-zod-client` output — adjust wrappers accordingly):

```ts
// Auto-generated from OpenAPI specs. Do not edit manually.
import { z } from 'zod';

// From MAPI components spec
export const ComponentSchemaField = z.object({ ... });
export const Component = z.object({ ... });

// From MAPI stories spec
export const Story = z.object({ ... });

// From MAPI assets spec
export const Asset = z.object({ ... });

// From MAPI datasources spec
export const Datasource = z.object({ ... });

// From CAPI stories spec (value shapes in story content)
export const AssetField = z.object({ ... });
export const MultilinkField = z.object({ ... });
export const RichtextField = z.object({ ... });
export const TableField = z.object({ ... });
export const PluginField = z.object({ ... });
export const StoryContent = z.object({ ... });
```

Known issues to address via post-processing in `generate.ts`:
- `Component.schema` uses `additionalProperties: true` → becomes `z.record(z.string(), z.unknown())`. Fine for the generated output; the precise field discriminated union is handled manually in `src/zod/schemas/field-schema.ts`.
- `ComponentSchemaField.type` is `z.string()` (no enum in OpenAPI) → post-process to `z.enum([...all field types...])`.
- `StoryContent` schema uses `additionalProperties` with `oneOf` → likely outputs `z.record(...)` or similar; fine for the generated schema. (Note: After Part 0, the MAPI spec uses the shared `story-content.yaml` instead of the old `blok.yaml`, so both MAPI and CAPI produce a `StoryContent` schema with the same field-type discriminators.)

## 3.3 Field schema — `src/zod/schemas/field-schema.ts`

A discriminated union Zod schema for validating field definitions (the `defineField` input at runtime). This is the Zod counterpart of `Field<TFieldType>` from Part 2.

```ts
import { z } from 'zod';
import type { FieldType } from '../../types/field';

// Base props shared across all field types
const baseFieldProps = {
  display_name: z.string().optional(),
  description: z.string().optional(),
  translatable: z.boolean().optional(),
  required: z.boolean().optional(),
  regex: z.string().optional(),
  default_value: z.string().optional(),
  pos: z.number().int().optional(),
};

// Per-type config schemas — must be strict() to reject unknown keys
const textFieldSchema = z.object({
  type: z.literal('text'),
  ...baseFieldProps,
  maxlength: z.number().int().optional(),
  minlength: z.number().int().optional(),
  rtl: z.boolean().optional(),
  rich_markdown: z.boolean().optional(),
}).strict();

const textareaFieldSchema = z.object({
  type: z.literal('textarea'),
  ...baseFieldProps,
  maxlength: z.number().int().optional(),
  minlength: z.number().int().optional(),
  rtl: z.boolean().optional(),
}).strict();

const richtextFieldSchema = z.object({
  type: z.literal('richtext'),
  ...baseFieldProps,
  restrict_components: z.boolean().optional(),
  component_whitelist: z.array(z.string()).optional(),
  customize_toolbar: z.boolean().optional(),
}).strict();

const markdownFieldSchema = z.object({
  type: z.literal('markdown'),
  ...baseFieldProps,
  rich_markdown: z.boolean().optional(),
  rtl: z.boolean().optional(),
}).strict();

const numberFieldSchema = z.object({
  type: z.literal('number'),
  ...baseFieldProps,
  minimum: z.number().optional(),
  maximum: z.number().optional(),
  decimals: z.number().int().optional(),
  steps: z.number().optional(),
}).strict();

const datetimeFieldSchema = z.object({
  type: z.literal('datetime'),
  ...baseFieldProps,
  disable_time: z.boolean().optional(),
}).strict();

const booleanFieldSchema = z.object({
  type: z.literal('boolean'),
  ...baseFieldProps,
  inline_label: z.string().optional(),
}).strict();

const optionSchema = z.object({
  _uid: z.string().optional(),
  name: z.string(),
  value: z.string(),
});

const optionFieldSchema = z.object({
  type: z.literal('option'),
  ...baseFieldProps,
  options: z.array(optionSchema).optional(),
  source: z.enum(['self', 'internal', 'external']).optional(),
  datasource_slug: z.string().optional(),
  exclude_empty_option: z.boolean().optional(),
  include_empty_option: z.boolean().optional(),
}).strict();

const optionsFieldSchema = z.object({
  type: z.literal('options'),
  ...baseFieldProps,
  options: z.array(optionSchema).optional(),
  source: z.enum(['self', 'internal', 'external']).optional(),
  datasource_slug: z.string().optional(),
  minimum_entries: z.number().int().optional(),
  maximum_entries: z.number().int().optional(),
}).strict();

const assetFieldSchema = z.object({
  type: z.literal('asset'),
  ...baseFieldProps,
  filetypes: z.array(z.string()).optional(),
  folder_slug: z.string().optional(),
  restrict_assets: z.boolean().optional(),
  asset_whitelist: z.array(z.string()).optional(),
  allow_external_url: z.boolean().optional(),
}).strict();

const multiassetFieldSchema = z.object({
  type: z.literal('multiasset'),
  ...baseFieldProps,
  filetypes: z.array(z.string()).optional(),
  folder_slug: z.string().optional(),
  restrict_assets: z.boolean().optional(),
  asset_whitelist: z.array(z.string()).optional(),
  minimum_entries: z.number().int().optional(),
  maximum_entries: z.number().int().optional(),
}).strict();

const multilinkFieldSchema = z.object({
  type: z.literal('multilink'),
  ...baseFieldProps,
  restrict_content_types: z.boolean().optional(),
  component_whitelist: z.array(z.string()).optional(),
  allow_target_blank: z.boolean().optional(),
  force_link_scope: z.boolean().optional(),
}).strict();

const bloksFieldSchema = z.object({
  type: z.literal('bloks'),
  ...baseFieldProps,
  restrict_components: z.boolean().optional(),
  component_whitelist: z.array(z.string()).optional(),
  component_group_whitelist: z.array(z.string()).optional(),
  minimum: z.number().int().optional(),
  maximum: z.number().int().optional(),
  restrict_type: z.string().optional(),
}).strict();

const tableFieldSchema = z.object({
  type: z.literal('table'),
  ...baseFieldProps,
}).strict();

const sectionFieldSchema = z.object({
  type: z.literal('section'),
  ...baseFieldProps,
  keys: z.array(z.string()).optional(),
}).strict();

const tabFieldSchema = z.object({
  type: z.literal('tab'),
  ...baseFieldProps,
  keys: z.array(z.string()).optional(),
}).strict();

const customFieldSchema = z.object({
  type: z.literal('custom'),
  ...baseFieldProps,
  field_type: z.string().optional(),
  options: z.unknown().optional(),
}).strict();

/**
 * Discriminated union Zod schema for all field types.
 * Use this to validate a field definition at runtime.
 * Also useful for composing into larger schemas.
 */
export const fieldSchema = z.discriminatedUnion('type', [
  textFieldSchema,
  textareaFieldSchema,
  richtextFieldSchema,
  markdownFieldSchema,
  numberFieldSchema,
  datetimeFieldSchema,
  booleanFieldSchema,
  optionFieldSchema,
  optionsFieldSchema,
  assetFieldSchema,
  multiassetFieldSchema,
  multilinkFieldSchema,
  bloksFieldSchema,
  tableFieldSchema,
  sectionFieldSchema,
  tabFieldSchema,
  customFieldSchema,
]);

export type FieldSchema = z.infer<typeof fieldSchema>;
```

Note: These schemas are hand-written (not generated) because the OpenAPI `ComponentSchemaField` schema uses a plain `string` for `type` without an enum, and the discriminated union structure cannot be auto-generated. They should be treated as the source of truth for field validation config shapes.

## 3.4 Prop schema — `src/zod/schemas/prop-schema.ts`

Extends each field schema variant with component-level prop configuration:

```ts
import { z } from 'zod';
import { fieldSchema } from './field-schema';

// Component-level prop config — these keys are added on top of the field config
const propConfigSchema = z.object({
  pos: z.number().int().optional(),
  required: z.boolean().optional(),
});

/**
 * Zod schema for a field as configured within a component schema.
 * Merges field schema + prop config.
 * Note: z.discriminatedUnion cannot be directly merged; use intersection instead.
 */
export const propSchema = fieldSchema.and(propConfigSchema);

export type PropSchema = z.infer<typeof propSchema>;
```

Note: `z.merge()` is deprecated in Zod v3. Use `.and()` (intersection) for merging with a discriminated union. In Zod v4 (if/when migrated), this approach may need updating.

### `src/zod/parsers/parse-story.ts`

This is the most complex parser. It dynamically constructs a Zod schema from the component definition and uses it to validate story content.

```ts
import { z } from 'zod';
import { contentValueSchemas } from '../schemas/content-value-schemas';
import type { Component, ComponentSchema } from '../../types/component';
import type { Story, StoryContent } from '../../types/story';
import type { FieldType } from '../../types/field';

export interface ParseStoryOptions {
  /**
   * If true, fills missing optional fields with their `default_value`
   * from the component schema before validation.
   */
  withDefaults?: boolean;
}

/**
 * Dynamically validates story content against a component schema at runtime.
 * Throws ZodError if:
 * - A field value has the wrong type
 * - A required field is missing
 * - Extra keys are present in content (strict mode)
 *
 * Automatically injects `component: component.name` into the returned story.
 *
 * @example
 * const story = parseStory(heroComponent, {
 *   content: { headline: 'Hello' }
 * }, { withDefaults: true });
 */
export const parseStory = <
  TName extends string,
  TSchema extends ComponentSchema,
>(
  component: Component<TName, TSchema>,
  story: Omit<Story<Component<TName, TSchema>>, 'content'> & {
    content: Omit<StoryContent<Component<TName, TSchema>>, 'component' | '_uid'>;
  },
  options: ParseStoryOptions = {},
): Story<Component<TName, TSchema>> => {
  // 1. Build the content Zod schema dynamically from the component's schema
  const shape: Record<string, z.ZodTypeAny> = {};

  for (const [key, prop] of Object.entries(component.schema)) {
    const fieldType = prop.type as FieldType;
    let valueSchema = contentValueSchemas[fieldType];

    // Handle withDefaults: pre-fill missing values before validation
    if (options.withDefaults && prop.default_value !== undefined) {
      valueSchema = valueSchema.default(prop.default_value);
    }

    // Make optional if not explicitly required
    if (!prop.required) {
      valueSchema = valueSchema.optional();
    }

    // Exclude layout-only fields (section, tab) from content shape
    if (fieldType !== 'section' && fieldType !== 'tab') {
      shape[key] = valueSchema;
    }
  }

  // 2. Strict object: reject extra keys not defined in the component schema
  const contentSchema = z.object(shape).strict();

  // 3. Apply withDefaults transformation before parsing
  let inputContent: Record<string, unknown> = { ...story.content as Record<string, unknown> };

  if (options.withDefaults) {
    for (const [key, prop] of Object.entries(component.schema)) {
      if (!(key in inputContent) && prop.default_value !== undefined) {
        inputContent[key] = prop.default_value;
      }
    }
  }

  // 4. Validate content
  const parsedContent = contentSchema.parse(inputContent);

  // 5. Return fully typed story with injected component name
  return {
    ...story,
    content: {
      ...parsedContent,
      component: component.name,
    },
  } as Story<Component<TName, TSchema>>;
};
```

## Key design decisions

### Why `fieldSchema` is hand-written, not generated

The OpenAPI `ComponentSchemaField` schema does not use a discriminated union on `type` — it defines `type` as a plain string with all possible config keys in a single flat object. This cannot produce a proper discriminated union Zod schema from `openapi-zod-client`. The hand-written discriminated union in `src/zod/schemas/field-schema.ts` is therefore the source of truth for field validation, not the generated output.

### Why `z.discriminatedUnion` over `z.union`

Discriminated unions in Zod give significantly better error messages when validation fails — they immediately identify which variant was attempted rather than reporting all failed variants. Since the `type` key always discriminates Storyblok field types, `z.discriminatedUnion('type', [...])` is the correct choice.

### `withDefaults` implementation approach

The `withDefaults` option pre-fills `default_value` from the component schema into the input content before validation. This happens at the JS object level, not via Zod's `.default()` modifier, because the default values come from the component schema definition (not the Zod schema itself). The `z.ZodDefault` approach via `.default(prop.default_value)` is used as a secondary mechanism in the `contentSchema` shape for completeness.

### Strict vs. non-strict story content

`z.object(shape).strict()` is used for story content validation to prevent extra keys. This is intentional — if a component schema changes and a key is removed, existing stories would fail validation until they are cleaned up. If this proves too strict for migration use cases, consider making strict mode opt-in via a `ParseStoryOptions.strict` flag.

### Zod version compatibility

The implementation targets Zod v3. The following Zod v3 caveats apply:
- `z.merge()` is deprecated — use `.and()` (intersection) for merging schemas.
- `z.discriminatedUnion()` does not support `.and()` on the union itself for adding shared props — use per-variant `.extend()` or spread `baseFieldProps` into each variant's `.object()`.
- If Zod v4 is adopted in the future, review the `propSchema` approach since the `.and()` intersection behavior may change.
